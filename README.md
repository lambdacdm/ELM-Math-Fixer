# ELM Math Fixer

ELM Math Fixer is a small Chrome extension for fixing common KaTeX rendering failures on the University of Edinburgh ELM platform. It is designed to be used together with a short prompt rule that prevents AI-generated display formulas from being split by Markdown before KaTeX can render them.

Generated by Claude Sonnet 5 and Gemini Flash 3.5.

## What It Fixes

ELM appears to run Markdown parsing before KaTeX rendering. That can break math in two common ways:

- Display formulas are split across Markdown paragraphs when `$$...$$` contains literal line breaks.
- Subscript underscores such as `x_{ij}` may be interpreted as Markdown emphasis markers and removed before KaTeX sees the formula.

The extension rescues affected math blocks in the browser. The prompt below prevents the line-break failure at generation time.

## Install Locally in Chrome

1. Download or clone this repository.
2. Open `chrome://extensions`.
3. Enable **Developer mode**.
4. Click **Load unpacked**.
5. Select the `ELM-Math-Fixer` folder.
6. Open or refresh `https://elm.edina.ac.uk/`.

To update after editing the files, return to `chrome://extensions` and click the reload button on the ELM Math Fixer card.

## ELM Prompt

Add this prompt to ELM as a system/custom instruction, or paste it at the beginning of a conversation:

```text
When generating mathematical formulas ($...$ or $$...$$), you must follow these rules, or the formula will fail to render or won't be recognized at all:

1. [MOST IMPORTANT] Never include a line break or blank line inside a $$...$$ formula. Everything from the opening $$ to the closing $$ must be one continuous, unbroken block of text: no line breaks in between, even for long formulas; keep it all on the same line/paragraph. Otherwise the platform will split the formula partway through and it won't render at all. If you need multi-line display, use an aligned/array/gathered environment with \\ line breaks inside the LaTeX itself, never insert a literal line break at the text level.

2. Whenever the character following ^ (superscript) or _ (subscript) is not a single letter or digit, it must be wrapped in curly braces, including control sequences themselves:
   Write \rho^{\#}, x_{ij}, \sigma^{-1}; do not write \rho^\#, x_ij, \sigma^-1.
```

## Package for Chrome Web Store

Create a ZIP containing the extension files at the repository root. The ZIP must contain `manifest.json` at its top level, not inside an extra parent folder.

```powershell
Compress-Archive -Path manifest.json,content.js,katex,icons,README.md,STORE_SUBMISSION.md,THIRD_PARTY_NOTICES.md -DestinationPath ELM-Math-Fixer-v1.0.zip -Force
```

Then follow the Chrome Web Store submission checklist in `STORE_SUBMISSION.md`.

## 这个插件解决什么

ELM 平台看起来是先做 Markdown 渲染，再做 KaTeX 公式渲染。这会导致两类常见问题：

- 如果 `$$...$$` 展示公式内部出现真实换行，公式会被 Markdown 拆成多个段落，KaTeX 无法识别完整公式。
- `x_{ij}` 这类下标里的 `_` 可能被 Markdown 当成斜体标记吃掉，导致 KaTeX 看到的公式已经损坏。

浏览器插件负责在网页端抢救已经被 Markdown 破坏的公式。下面的提示词负责从生成端避免公式内换行问题。

## 在 Chrome 本地安装

1. 下载或克隆这个仓库。
2. 打开 `chrome://extensions`。
3. 打开右上角 **开发者模式**。

## ELM 提示词

把这段加入 ELM 的 system prompt / custom instruction，或者每次对话开始时粘贴：

```text
生成数学公式（$...$ 或 $$...$$）时，必须遵守以下规则，否则公式会渲染失败或完全不被识别：

1.【最重要】一条 $$...$$ 公式内部绝对不能换行或有空行。从开头 $$ 到结尾 $$ 之间必须是连续的一整段文本，中间不能敲回车，哪怕公式很长也要写在同一行/同一段落里，否则平台会把公式从中间切断，导致完全不渲染。需要分行展示时，用 aligned/array/gathered 环境配合 \\ 处理，不要在文本层面换行。

2. 上标 ^ 和下标 _ 后面只要不是单个字母/数字，一律加花括号，包括控制序列本身：
   写 \rho^{\#}、x_{ij}、\sigma^{-1}，不要写 \rho^\#、x_ij、\sigma^-1。
```
